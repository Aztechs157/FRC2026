## Build & Deploy Commands

```bash
./gradlew build          # Compile the project
./gradlew deploy         # Deploy to the RoboRIO over USB/WiFi
./gradlew downloadAll    # Download FRC tools and libraries
./gradlew simulateJava   # Run robot code in simulation (opens GUI)
./gradlew replayWatch    # AdvantageKit log replay
```

There is no test suite ‚Äî this is a robotics control system where testing happens on hardware or in simulation.

## Architecture Overview

**Core pattern:** WPILib Command-Based framework + AdvantageKit IO abstraction layer.

Every subsystem follows this structure:
```
FooSubsystem (extends SubsystemBase)
  ‚îú‚îÄ‚îÄ FooIO (interface) ‚Äî defines inputs struct + hardware methods
  ‚îÇ     ‚îú‚îÄ‚îÄ FooIOTalonFX ‚Äî real robot hardware (Phoenix 6)
  ‚îÇ     ‚îî‚îÄ‚îÄ FooIOSim ‚Äî simulation implementation
  ‚îî‚îÄ‚îÄ FooIOInputsAutoLogged ‚Äî generated by AdvantageKit annotation processor
```

The IO implementation is injected at construction time in `RobotContainer` based on `Robot.isReal()`. This means all robot I/O is mocked-out in sim by default.

**Entry points:**
- `Main.java` ‚Üí `Robot.java` (extends `LoggedRobot`) ‚Üí `RobotContainer.java`
- `Robot.java` initializes the AdvantageKit logger, sets logging mode (real/sim/replay), and starts the CommandScheduler
- `RobotContainer.java` instantiates all subsystems, wires button bindings, and registers PathPlanner named commands

## Key Frameworks & Libraries

- **WPILib 2026.1.1** ‚Äî Command-based robot framework
- **Phoenix 6 (26.1.1)** ‚Äî CTRE TalonFX motor controllers and Pigeon2 IMU; uses CANivore bus for drive
- **AdvantageKit** ‚Äî All subsystem inputs use `@AutoLog`; real robot logs to WPILOGWriter (USB), sim publishes via NT4; replay is supported via `HootAutoReplay`
- **PathPlanner v2026.1.2** ‚Äî Autonomous paths in `src/main/deploy/pathplanner/`; named commands (e.g. `DeployIntake`, `ShootBalls`) must be registered in `RobotContainer` before use
- **PhotonLib** ‚Äî Vision with multi-tag PnP pose estimation; camera transforms defined in `VisionConstants`
- **Yet Another Mechanism System (2026.2.8)** -- Hardware abstraction framework for mechanism control & simulation

## AdvantageKit Conventions

- IO input classes are annotated with `@AutoLog`; the `*IOInputsAutoLogged` class is **generated** ‚Äî do not edit it manually
- Call `Logger.processInputs("SubsystemName", inputs)` in each subsystem's `periodic()` to record inputs
- `AKitTunerConstants.java` in `generated/` is also auto-generated by PhoenixTuner ‚Äî do not edit manually
- `BuildConstants.java` is generated at build time

## Constants Organization

All constants live in `Constants.java` as nested static classes:
- `ModifierConstants` ‚Äî Demo/Rookie/Maya mode toggles and drive speed multipliers
- `ControllerConstants` ‚Äî Port numbers and joystick deadband (0.05)
- `VisionConstants` ‚Äî Camera names and 3D robot-to-camera transforms
- `FieldConstants` ‚Äî Field dimensions and manual reset positions
- `ModelConstants` ‚Äî 3D offsets for AdvantageScope mechanism visualization

Subsystem-specific constants (PID gains, gear ratios, angle limits) live within each subsystem's own file (e.g., `FlywheelConstants` in `Flywheel.java`).

## Autonomous

Autonomous routines are PathPlanner paths stored in `src/main/deploy/pathplanner/autos/`. Named commands used in paths must be registered in `RobotContainer.java` via `NamedCommands.registerCommand(...)` before `AutoBuilder` is configured.

The auto selector uses `SendableChooser` on SmartDashboard. PathPlanner's `AutoBuilder.buildAutoChooser()` populates it automatically from the deploy directory.
